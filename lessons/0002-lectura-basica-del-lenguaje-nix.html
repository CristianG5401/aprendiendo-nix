Lección 0002 · Lenguaje Nix mínimo

Leer Nix sin miedo: let, sets y listas

Objetivo: entender la sintaxis mínima que ya apareció en tu shell.nix, usando comparaciones con JavaScript/TypeScript.

Por qué esta lección existe

Completaste la lección 0001 y apareció una duda correcta: let ... in. Antes de avanzar a flake.nix, vamos a calibrar el auspex sintáctico: Nix no es Bash, no es JSON puro y tampoco es TypeScript, pero puedes leerlo con herramientas mentales parecidas.

Regla del curso desde ahora: no asumiremos que sabes el lenguaje Nix. Cada símbolo nuevo será explicado.

1. Nix evalúa expresiones, no ejecuta pasos

En JavaScript/TypeScript estás acostumbrado a instrucciones:

const x = 1;
const y = 2;
console.log(x + y);

En Nix normalmente escribes una expresión que produce un valor:

let
  x = 1;
  y = 2;
in
  x + y

Resultado:

3

Fuente: nix.dev explica que un archivo .nix contiene una expresión Nix, y evaluarla produce un valor.

2. let ... in: variables locales para una expresión

Forma general:

let
  nombre = valor;
  otroNombre = otroValor;
in
  expresionFinal

Analogía TypeScript:

(() => {
  const nombre = valor;
  const otroNombre = otroValor;
  return expresionFinal;
})()

Ejemplo Nix:

let
  tool = "jq";
  purpose = "procesar JSON";
in
  "${tool} sirve para ${purpose}"

Equivalente mental en TS:

(() => {
  const tool = "jq";
  const purpose = "procesar JSON";
  return `${tool} sirve para ${purpose}`;
})()

3. Attribute sets: objetos tipo JSON, pero con = y ;

En Nix, un attribute set se parece mucho a un objeto JS/JSON:

{
  hostname = "homeserver";
  sshPort = 22;
  monitoring = true;
}

Analogía JS/TS:

{
  hostname: "homeserver",
  sshPort: 22,
  monitoring: true,
}

Diferencias importantes:

  • Nix usa = en vez de :.
  • Nix termina cada atributo con ;, no con coma.
  • Accedes con punto: server.hostname.
{ hostname = "homeserver"; }.hostname

Resultado:

"homeserver"

4. Listas: arrays sin comas

En Nix, las listas se escriben con espacios, no con comas:

[ "curl" "git" "jq" "htop" ]

Analogía TS:

["curl", "git", "jq", "htop"]

En tu shell.nix, esta lista no contiene strings, contiene paquetes:

packages = with pkgs; [
  curl
  git
  jq
  htop
];

Lectura humana:

Dentro del conjunto pkgs, toma los paquetes curl, git, jq y htop.

5. Releer tu shell.nix con ojos nuevos

let
  nixpkgs = fetchTarball "https://github.com/NixOS/nixpkgs/tarball/nixos-24.05";
  pkgs = import nixpkgs { config = {}; overlays = []; };
in

pkgs.mkShellNoCC {
  packages = with pkgs; [
    curl
    git
    jq
    htop
  ];

  LAB_NAME = "home-lab-reproducible";
}

Traducción ceremonial:

  1. nixpkgs: fuente de paquetes a usar.
  2. pkgs: colección importada desde esa fuente.
  3. pkgs.mkShellNoCC: función que crea una shell sin toolchain C.
  4. El bloque { ... }: configuración/argumento para esa shell.
  5. packages = [ ... ];: lista de herramientas disponibles dentro de la shell.

Ejercicio guiado: crea un archivo Nix que solo evalúa datos

Crea este archivo:

mkdir -p ~/estudio/linux/nix/lab/lesson-0002
cd ~/estudio/linux/nix/lab/lesson-0002
nano basics.nix

Pega:

let
  server = {
    hostname = "homeserver";
    role = "lab";
    tools = [ "curl" "git" "jq" ];
  };
in
{
  message = "${server.hostname} es un servidor de tipo ${server.role}";
  inherit (server) tools;
}

Evalúalo:

nix-instantiate --eval --strict basics.nix

Resultado esperado:

{ message = "homeserver es un servidor de tipo lab"; tools = [ "curl" "git" "jq" ]; }

Si nix-instantiate no existe, dime el error exacto. No haremos conjuros a ciegas.

Práctica de recuperación

Sin mirar arriba: en Nix, ¿qué dos signos reemplazan normalmente a : y , cuando escribes un objeto estilo JSON?

Nix usa = para asignar atributos y ; para terminar cada atributo. Ejemplo: { name = "server"; }
