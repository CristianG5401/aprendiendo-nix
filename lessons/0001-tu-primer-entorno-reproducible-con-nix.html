Lección 0001 · Nix para home labs

Tu primer entorno reproducible con Nix

Objetivo: describir un conjunto de herramientas CLI en un archivo versionable, no en tu memoria.

La idea central

En un servidor tradicional, solemos instalar herramientas con comandos imperativos: apt install jq curl git. Eso funciona, pero meses después quizá no recuerdas exactamente qué instalaste ni por qué. Nix permite mover parte de ese conocimiento a un archivo declarativo.

En esta primera liturgia no vamos a migrar el servidor a NixOS. Solo usaremos Nix como herramienta para entrar en un entorno reproducible dentro de Ubuntu Server.

Vocabulario mínimo

  • Nix: package manager y lenguaje para describir paquetes/entornos.
  • Nixpkgs: colección enorme de paquetes disponibles para Nix.
  • shell reproducible: terminal temporal con herramientas declaradas en un archivo.
  • shell.nix: archivo que describe qué herramientas entran en la shell.

Ejercicio: crea una shell de herramientas de servidor

Crea una carpeta para practicar:

mkdir -p ~/nix-lab/lesson-0001
cd ~/nix-lab/lesson-0001

Dentro, crea este archivo shell.nix:

let
  nixpkgs = fetchTarball "https://github.com/NixOS/nixpkgs/tarball/nixos-24.05";
  pkgs = import nixpkgs { config = {}; overlays = []; };
in

pkgs.mkShellNoCC {
  packages = with pkgs; [
    curl
    git
    jq
    htop
  ];

  LAB_NAME = "home-lab-reproducible";

  shellHook = ''
    echo "Entorno Nix activo: $LAB_NAME"
    echo "Herramientas disponibles: curl git jq htop"
  '';
}

Entra al entorno:

nix-shell

Verifica que las herramientas están ahí:

which jq
jq --version
echo "$LAB_NAME"

Qué está pasando

  • fetchTarball fija una rama de Nixpkgs: aquí usamos nixos-24.05.
  • import nixpkgs carga esa colección de paquetes.
  • pkgs.mkShellNoCC crea una shell sin meter un compilador innecesario.
  • packages añade herramientas al PATH dentro de esa shell.
  • LAB_NAME se convierte en environment variable.
  • shellHook ejecuta comandos al entrar en la shell.

Práctica de recuperación

Antes de mirar la respuesta: ¿qué problema resuelve shell.nix frente a recordar comandos apt install?

Convierte el entorno en un archivo declarativo y versionable. En vez de depender de memoria o notas sueltas, puedes copiar o clonar ese archivo y reconstruir la misma shell en otra máquina.

Tu pequeña victoria

Si este ejercicio funciona, ya tienes el primer ladrillo para tu objetivo: una forma de describir herramientas de administración de servidores sin depender del estado accidental de una máquina concreta.

Siguiente paso sugerido: aprender la sintaxis mínima del lenguaje Nix antes de añadir mecanismos más avanzados de reproducibilidad.
